    功能¶

    在 C++ 中，#define 是一個預處理指令，用來在編譯階段之前定義宏（macros）。#define 可以用於多種不同的情境，以下是一些主要的用法：

    1. 定義常數： #define 可以用來創建編譯時常數。這種用法可以代替 const 或 constexpr，特別是在需要保證值在編譯時就是已知的場合。
    #define MAX_SIZE 100
#define COMPANY_NAME "Example Corp."

    1. 創建宏函數： 宏可以被用來創建類似函數的結構，但它們不涉及函數調用的開銷，因為它們在編譯時就被展開。

      注意變數在運算時要注意先加減後乘除問題，如果#define SQUARE(x) ((x) * (x))，傳入x=2+1，如果沒有括號，會變成(2+1*2+1)，答案為5，不是3*3=9

    #define SQUARE(x) ((x) * (x))
#define MIN(a, b) ((a) < (b) ? (a) : (b))
    1. 條件編譯： #define 可用於控制哪些代碼段被編譯。這常用於依據不同的編譯環境或配置來包含或排除特定代碼。
    #define DEBUG

#ifdef DEBUG
std::cout << "Debug mode is on." << std::endl;
#endif
    1. 防止頭文件重複包含： 通過使用宏來防止頭文件被多次包含，這是預防編譯器多次處理同一頭文件的常用技術。
    #ifndef MY_HEADER_H
#define MY_HEADER_H
// 頭文件內容
#endif
    1. 平台特定代碼： 使用 #define 來根據不同操作系統或硬件配置啟用或禁用代碼塊。
    #ifdef WINDOWS
#define OS_NAME "Windows"
#elif defined(LINUX)
#define OS_NAME "Linux"
#else
#define OS_NAME "Unknown"
#endif
    1. 簡化複雜的代碼： 有時 #define 用來簡化複雜或冗長的代碼結構，使其更容易閱讀和管理。
    #define LOG_ERROR(msg) std::cerr << "Error: " << msg << std::endl

    每種用法都有其特定的場景和考慮因素。使用 #define 需要謹慎，因為它只進行文本替換，不進行類型檢查，且可能導致難以追蹤的錯誤。在現代 C++ 中，通常推薦使用更安全的替代方案，如 const、constexpr 和模板，這些都提供更好的類型安全和錯誤檢查。

    優缺點¶

    在 C++ 中，選擇使用宏函數還是直接創建函數，各有其優缺點，以下是比較兩者的主要因素：

    宏函數 (使用 #define)¶

    優點：

    1. 無額外調用開銷：宏是在編譯期進行文本替換，所以使用宏可以避免函數調用的開銷，特別是在微優化是重要的場景中。
    2. 類型獨立：宏可以用於任何類型的數據，因為它們進行的是文本替換，而不檢查類型。
    3. 條件編譯：可以在宏中嵌入預處理器指令，使得代碼的某些部分只在特定條件下編譯。

    缺點：

    1. 易於引起錯誤：宏不進行類型檢查，且因為文本替換的特性，可能導致如運算優先級等錯誤。
    2. 調試困難：宏展開後的代碼可能很難理解和調試，因為它們在編譯時展開，錯誤定位通常不直觀。
    3. 代碼膨脹：每次使用宏時，代碼都會在該處展開，可能導致生成的二進制文件增大。

    直接創建函數¶

    優點：

    1. 類型安全：函數提供類型檢查，減少類型錯誤的風險。
    2. 易於維護和調試：函數的行為更清晰，易於維護和調試，因為它們在編譯後保持結構清晰。
    3. 重用和封裝：函數允許代碼重用和更好的封裝，提高了代碼的模組化。

    缺點：

    1. 可能有額外開銷：每次調用函數時可能有額外的開銷（如堆疊操作），但在現代編譯器優化和 inline 函數的幫助下，這通常不是問題。
    2. 不如宏靈活：函數無法像宏那樣進行條件編譯或包含預處理器指令。

    總結¶

    在現代 C++ 程序設計中，除非有特定的理由（如微優化或需要在宏中使用預處理器指令），否則通常推薦使用函數而不是宏。函數提供更好的類型安全、易於維護和調試。使用 inline 函數可以結合宏的性能優勢和函數的安全特性，通常是一種更好的選擇。
